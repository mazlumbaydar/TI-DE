Background

What Happened

On 31 March 2026, security researchers at StepSecurity disclosed that two versions of the axios npm package — one of the most downloaded JavaScript libraries with over 60 million weekly downloads — had been quietly replaced with malicious builds. The compromised versions were axios@1.14.1 and axios@0.30.4. The legitimate previous versions (1.14.0 and 0.30.3) remained unaffected.

The attack worked by injecting a dependency on plain-crypto-js@4.2.1 into the malicious axios builds. This package executed an obfuscated postinstall hook (node setup.js) the moment any developer ran npm install. The hook detected the host operating system and dropped a platform-specific RAT payload, then erased its own traces.

Safe versions: axios@1.14.0 and axios@0.30.3 are clean. Versions axios@1.14.1 and axios@0.30.4 are malicious — remove immediately and audit any environment that had these versions installed.
Attack Flow

Attack Chain

📦
Stage 1 — Poisoned Package Published
Attacker gains access to the axios npm account and publishes axios@1.14.1 and axios@0.30.4. Both versions add a hidden dependency: plain-crypto-js@4.2.1, which carries the malicious postinstall hook. The package diff is minimal and passes casual visual inspection.
T1195.001 — Compromise Software Supply Chain
⚙️
Stage 2 — Postinstall Hook Executes
When the developer runs npm install, npm executes the postinstall script: node setup.js. The script uses a custom XOR decoder with key OrDeR_7077, index formula 7*i*i % 10, and constant 333 to reconstruct the payload URLs from reversed, padded base64 strings — bypassing static signature scanners that rely on readable URLs.
T1140 — Deobfuscate/Decode Files or Information
🖥️
Stage 3 — OS Fingerprinting
The setup.js script calls os.platform() to branch execution: win32 → VBScript dropper, darwin → macOS LaunchAgent with AppleScript, linux → Python3 script in /tmp with nohup persistence. Each branch uses a different stealth technique suited to its target OS.
T1059.001 / T1059.004 / T1059.006
💉
Stage 4 — Payload Delivery & Masquerade
Windows: VBScript spawns PowerShell to download a binary to %PROGRAMDATA%\wt.exe — masquerading as the legitimate Windows Terminal executable.
macOS: Binary copied to /Library/Caches/com.apple.act.mond — spoofing an Apple system daemon. AppleScript used for execution.
Linux: Python3 script written to /tmp/ld.py, launched via nohup to survive terminal closure.
T1036.005 — Match Legitimate Name or Location
📡
Stage 5 — C2 Beacon
The RAT establishes command-and-control with a 60-second polling interval. It uses a fake User-Agent string mozilla/4.0 (compatible; msie 8.0; windows nt 5.1; trident/4.0) to blend with legacy enterprise traffic. Supported commands include peinject, runscript, rundir, and kill.
T1071.001 — Web Protocols C2
🧹
Stage 6 — Anti-Forensic Cleanup
After payload delivery, setup.js deletes itself via fs.unlink and replaces package.json inside node_modules with a clean stub, removing the postinstall field. This makes post-infection forensics harder — the install log may show nothing suspicious by the time the analyst looks.
T1070.004 — File Deletion
Obfuscation Technique

XOR Payload Decoding

The dropper did not store payload URLs as plaintext. Instead it used a custom XOR cipher combined with reversed, padding-altered base64 strings. Below is the reconstructed decode logic from the setup.js artifact:

setup.js — XOR decoder (reconstructed)
const xorKey = "OrDeR_7077"; // key extracted from artifact const CONSTANT = 333; function decode(encoded) { // 1. Reverse the string; replace '_' back to '=' padding const b64 = encoded.split('').reverse().join('').replace(/_/g, '='); const buf = Buffer.from(b64, 'base64'); // 2. XOR each byte using rolling key + position formula return Buffer.from( buf.map((byte, i) => { const keyChar = xorKey.charCodeAt((7 * i * i % 10) % xorKey.length); return byte ^ keyChar ^ (CONSTANT % 256); }) ).toString('utf8'); } // os.platform() branch selects Windows / Linux / macOS payload const platform = require('os').platform(); const payloadUrl = decode(ENCODED_URLS[platform] ?? ENCODED_URLS.linux);
The non-standard index formula 7*i*i % 10 and the reversed base64 with underscore padding substitution are strong behavioral signatures for YARA detection, as they are unlikely to appear in legitimate packages.

Response & Hardening

Detection & Remediation Recommendations

🔍
Audit Installed Packages
Run npm audit and check for axios@1.14.1 or axios@0.30.4 in all CI/CD environments, developer workstations, and Docker base images. Downgrade to 1.14.0 / 0.30.3 or upgrade past the fixed version.
🚫
Block Postinstall Network Access
Apply network policies that prevent npm scripts from initiating outbound connections. Consider --ignore-scripts flag in CI/CD pipelines or configure npm to disallow lifecycle scripts by default.
📋
Lock Dependency Trees
Commit and pin package-lock.json. Use integrity hash verification. Consider private npm mirrors or proxies (e.g. Verdaccio, JFrog Artifactory) to control which package versions reach developers.
🔎
Hunt for Active Compromise
Search for processes running from C:\ProgramData\ with non-system parents, Python scripts in /tmp/ with nohup parents, and unexpected binaries in /Library/Caches/com.apple.*.
🛡️
Enable Supply Chain Monitoring
Adopt tools like Socket.dev, Snyk, or GitHub's dependency review action to alert on new transitive dependencies added to popular packages before they land in production.
📡
Monitor for Fake User-Agents
Proxy/NGFW logs: alert on outbound HTTP with IE8 / MSIE 8.0 User-Agent strings from developer or build machines — no legitimate modern application should emit this UA string.
