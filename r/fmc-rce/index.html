Background

What Happened

On 28 March 2026, Cisco disclosed CVE-2026-20131, a critical unauthenticated remote code execution vulnerability in Cisco Secure Firewall Management Center (FMC). The flaw exists in FMC's internal Java-based management API: a serialized Java object sent via HTTP POST to a specific endpoint is deserialized without any authentication or class allowlisting, handing the attacker a fully functional root shell on the FMC appliance.

The Interlock ransomware group exploited this as a zero-day before Cisco's patch. Within hours of gaining root access, they disabled all managed IDS/IPS sensors via FMC's management plane, moved laterally through the network, exfiltrated sensitive data, and deployed ransomware. Because FMC is the single control plane for all managed Cisco firewalls, compromise of FMC effectively blinds the entire network security posture.

Patch immediately to FMC 7.4.2+. If patching is not immediately possible, block all external access to FMC management interfaces (TCP 443, 8305) at the perimeter. Treat any FMC appliance that was internet-exposed as potentially compromised.

Attack Flow

Attack Chain — 6 Steps

🔍
Step 1 — Zero-Day Discovery & Reconnaissance
Interlock identifies CVE-2026-20131 in FMC's Java management API before public disclosure. The vulnerability exists in the deserialization of Java objects in unauthenticated API endpoints. Attacker uses Shodan/Censys to enumerate internet-exposed FMC instances. The management interface (TCP 443) is frequently exposed for remote administration.
T1595.002 — Active Scanning: Vulnerability Scanning
🛠
Step 2 — Malicious HTTP POST (Exploit Delivery)
Attacker crafts a serialized Java payload using the Commons Collections gadget chain (ysoserial CommonsCollections6) and sends it via HTTP POST to the FMC API endpoint. No authentication token, session cookie, or credentials are required. FMC's Java runtime deserializes the object and executes the embedded OS command.
T1190 — Exploit Public-Facing Application T1059.007 — JavaScript/Java
⚠
Step 3 — Java Deserialization → Root Shell
The gadget chain triggers a Runtime.exec() call within the JVM, spawning a reverse shell process as root (FMC runs its Java services as root). Attacker receives the connection on a remote listener. Because FMC runs on a hardened CentOS-based appliance, standard EDR agents are typically absent — the shell runs undetected at the OS layer.
T1059.004 — Unix Shell T1203 — Exploitation for Client Execution
🔒
Step 4 — IDS/IPS Disabled via FMC Management Plane
With root access to FMC, attackers directly manipulate the FMC database (SFDataCorrelator, the central policy engine). They disable all IPS/IDS policies on all managed firewalls, effectively removing network detection across the entire environment. This is done via CLI commands on FMC itself — no API calls needed, no audit logs written to SIEM.
T1562.001 — Disable or Modify System Firewall / IDS T1078 — Valid Accounts (root)
🔁
Step 5 — Lateral Movement + Data Exfiltration
FMC stores credentials for all managed devices, network topology maps, and firewall rule configurations. Attackers harvest these and pivot laterally across the network — the FMC's trusted management VRF provides unfiltered access. Configuration files, network architecture data, and VPN credentials are exfiltrated to the attacker's C2 infrastructure.
T1021 — Remote Services T1041 — Exfiltration over C2
🔥
Step 6 — Ransomware Deployment (Interlock)
After establishing persistence and disabling defenses, Interlock deploys their FreeBSD/Linux ransomware binary on FMC and deploys Windows variants to lateral systems. FMC configuration database is encrypted — the entire managed firewall estate becomes unmanageable. Interlock uses double-extortion: encrypted systems + threat to publish exfiltrated network configuration data.
T1486 — Data Encrypted for Impact T1657 — Financial Theft / Extortion
