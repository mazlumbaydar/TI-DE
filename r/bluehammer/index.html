Background

What Happened

On 3 April 2026, a disgruntled security researcher operating under the alias Nightmare-Eclipse publicly dropped a functional Windows local privilege escalation exploit on GitHub. The vulnerability, named BlueHammer, targets a time-of-check to time-of-use (TOCTOU) race condition in Microsoft Defender's signature-update workflow. Microsoft had been notified through MSRC but the researcher — frustrated with the response process — chose full public disclosure.

BlueHammer chains four legitimate Windows components: Microsoft Defender's update process, the Volume Shadow Copy Service (VSS), the Cloud Files API, and opportunistic locks (oplocks). The exploit registers a fake Cloud Files sync root, identifies Defender by PID, then triggers a signature update. During the update, Defender creates a VSS snapshot before it holds exclusive control — the race window. The attacker intercepts this via an oplock, traverses the snapshot to reach the SAM hive, and extracts NTLM hashes. A final GUID-named temporary service escalates the process to NT AUTHORITY\SYSTEM.

As of April 13, 2026, no patch exists and no CVE has been assigned. Microsoft pushed a Defender signature update (Exploit:Win32/DfndrPEBluHmr.BB) that detects the original PoC binary — but because the exploit abuses legitimate component behavior, recompiling from slightly modified source bypasses the signature entirely. The Howler Cell team at Cyderes independently verified and fixed bugs in the original PoC, running the exploit end-to-end against fully-patched Windows 10 and 11 systems.

BlueHammer requires only local access — no admin, no UAC bypass needed. Any user account on an unmitigated Windows 10/11 host can escalate to SYSTEM. Treat all shared Windows endpoints as at-risk until Microsoft issues a patch.

Attack Flow

Attack Chain — 6 Steps

🛠
Step 1 — Register Fake Cloud Files Sync Root
Attacker registers a malicious Cloud Files sync root via the Windows Cloud Files API (CfRegisterSyncRoot). A callback is registered that targets MsMpEng.exe (Defender) by PID. This callback can selectively stall or freeze Defender's file I/O operations — the foundation of the race condition. No elevation required to call CfRegisterSyncRoot.
T1106 — Native API T1562.001 — Impair Defenses
🔄
Step 2 — Trigger Defender Signature Update
The exploit triggers a Windows Defender signature update. During this process, Defender calls VolSnap to create a VSS snapshot before acquiring exclusive control over it. This creates the TOCTOU window: Defender has created the snapshot but has not yet locked it down, leaving a brief period where a low-privileged process can race to access the snapshot contents.
T1490 — Inhibit System Recovery (VSS abuse)
🔓
Step 3 — Oplock Race + VSS Snapshot Interception
An opportunistic lock (oplock) is placed on the VSS snapshot path. When Defender accesses the snapshot, the oplock fires and hands control to the attacker's callback. The attacker uses NtQueryDirectoryObject to enumerate HarddiskVolumeShadowCopy* entries and intercepts the snapshot mid-access. This is the core TOCTOU exploitation stage — winning the race grants read access to the snapshot as if the attacker were the high-privileged Defender process.
T1106 — Native API
💾
Step 4 — SAM Hive Extraction
Through the intercepted VSS snapshot, the attacker traverses to Windows\System32\config\SAM. The SAM hive contains NTLM password hashes for all local accounts, including the local Administrator. Under normal circumstances, the SAM is locked by the OS and inaccessible to non-SYSTEM processes. The shadow copy path bypasses this lock entirely — the attacker reads the SAM hive to disk without any privileged API call.
T1003.002 — OS Credential Dumping: SAM
🔑
Step 5 — NTLM Hash Extraction
The extracted SAM hive is parsed offline using standard tools (e.g., impacket-secretsdump or mimikatz lsadump::sam) to recover NTLM hashes. No live LSASS interaction is required — the SAM parsing happens entirely from the file on disk. The local Administrator NTLM hash is then usable for pass-the-hash attacks against remote systems on the same network.
T1003.002 — SAM Credential Dumping T1078.003 — Local Accounts
⚡
Step 6 — GUID Service → SYSTEM + Pass-the-Hash
The final stage creates a short-lived Windows service with a random GUID name (svchost_{random-guid}) via CreateService / StartService. The service binary executes the attacker's payload as NT AUTHORITY\SYSTEM. The service is deleted immediately after execution to reduce forensic artifacts. With SYSTEM access and recovered NTLM hashes, pass-the-hash enables lateral movement across all hosts sharing the same local admin password hash.
T1068 — Exploitation for Privilege Escalation T1550.002 — Pass the Hash

Incident Response

Response Actions

🛡
Deploy LAPS Immediately
Enable Local Administrator Password Solution (LAPS) on all Windows 10/11 endpoints. LAPS randomizes the local admin password per machine — even if BlueHammer recovers NTLM hashes, pass-the-hash lateral movement fails because no two machines share the same hash. This is the single most impactful mitigation available before a patch exists.
🔓
Disable VSS on Non-Critical Endpoints
VSS (Volume Shadow Copy Service) is a required component of the BlueHammer exploit chain. On endpoints where VSS is not needed for backup: sc config VSS start= disabled and sc stop VSS. Verify with backup teams before disabling. On endpoints where VSS is required, ensure strict process isolation monitoring.
👁
Enable Sysmon Event 7 (Image Load)
Deploy or update your Sysmon configuration to enable Event ID 7 (Image Load) with a filter matching samlib.dll. This single log source catches BlueHammer's SAM credential dump stage with near-zero false positives. Alert immediately on any process other than lsass.exe loading samlib.dll.
🔌
Alert on Event 7045 — GUID Service Names
Create a SIEM alert for Windows System Event ID 7045 (New Service Installed) where the service name matches GUID format. This event is generated by default on all Windows systems — no additional logging required. No legitimate Windows component creates GUID-named services at runtime. Every hit is a true positive.
📊
Hunt for Historical Exploitation Artifacts
Run retrospective hunts across 30–90 days of endpoint telemetry using the queries in the Detection section. Focus on: (1) samlib.dll image loads, (2) GUID-named service 7045 events, (3) SyncRootManager registry writes, (4) %TEMP% files matching SAM hive artifacts. Correlate timing with Defender update events.
⚠
Do Not Rely on Defender Signature Detection Alone
Microsoft's Defender signature Exploit:Win32/DfndrPEBluHmr.BB detects only the original compiled PoC binary. Because BlueHammer abuses legitimate Windows component behavior — not any specific binary — recompiling from source bypasses the signature entirely. Behavioral detection (samlib.dll load, GUID services, VSS enumeration) is the only reliable coverage.
