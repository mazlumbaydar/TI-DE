Engineering Process

Detection Methodology

🎯
Principle 01
TTP Over IOC
Detections are behavioral — never hardcoded IPs or domains. Attacker infrastructure rotates in hours; adversary TTPs persist for years.
🔗
Principle 02
Minimum 2 AND Conditions
Every rule requires at least 2 correlated conditions. Single-signal rules produce noise. Correlation reduces false positives without sacrificing coverage.
📐
Principle 03
Max 2 Rules per Threat
Rule 1: process chain (parent → child spawn). Rule 2: artifact/file activity. Consolidation prevents alert fatigue and simplifies triage.
🔍
Principle 04
Platform Field Verification
All field names are verified against official documentation per platform. Cortex XDR rules are BIOC-compatible. No copy-paste without testing field validity.
⚡
Principle 05
KEV & Active Exploitation First
Priority: CISA KEV entries and actively exploited vulnerabilities. Detection published within 24 hours of reliable intelligence confirming active exploitation.
🌍
Principle 06
Open Source — TLP:WHITE
All rules are freely available on GitHub. No paywalls, no registration. The security community improves collectively when detection logic is shared openly.
Rule Standards

What Every Rule Must Have

✅Parent-Child Process Correlation
Rules target the spawning relationship, not just the child process. langflow → bash is a detection; bash alone is noise.
🚫No Specific Filenames as IOCs
Not wt.exe or ld.py — the signal is writer process + destination path pattern. Attackers rename files trivially; directory staging patterns are durable.
🔒Platform Capability Checks
Rules are only written if the platform can realistically collect the required telemetry. Cortex XDR BIOC uses event_type = ENUM.PROCESS — not joins.
📌MITRE ATT&CK Mapping
Every rule maps to at least one MITRE technique. Enables correlation with threat actor profiles, campaign tracking, and coverage gap analysis.
